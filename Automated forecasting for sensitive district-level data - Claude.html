Assessed pipeline efficiency and architected metadata-only improvements

Short answer: this is a solid, sensible architecture for district-scale sensitive data — and the right call to keep the LLM out of the data path entirely. But the current rules are fairly coarse for production use on government/administrative time series, and there's room to make the deterministic side smarter and to add a safe, metadata-only LLM assist layer if you want one. Here's the breakdown.

What's already right

  • Deterministic routing/scoring means no raw record ever needs to leave the machine — good default for sensitive data.
  • Rolling-origin backtesting (no shuffling) is the correct evaluation style for time series.
  • Complexity penalty + preference bonus is a reasonable anti-overfitting mechanism for small-N data.
  • Per-group (panel) routing means a district's sub-units aren't forced into one model.

Where it's likely to underperform at district scale

  1. Static, row-count-based shortlist rules. Whether a model is even considered depends on row counts and crude signal flags (has_exogenous), not on actual statistical structure (trend strength, seasonal strength, stationarity, noise level). Two datasets with the same row count can have very different forecastability.
  2. Fixed SARIMAX(1,1,1) isn't tuned — for irregular admin data this is often a poor fit; a small AIC-grid search is cheap and bounded.
  3. No domain calendar effects. District-level government/agriculture/health data is heavily driven by festivals, monsoon onset, fiscal year boundaries, election cycles, etc. None of that is in the feature set — this is probably the single biggest accuracy gap for Indian administrative data.
  4. Only 3 backtest splits — for short series, this is noisy; weighting recent folds more, or using an expanding window with more folds when data allows, would stabilize model choice.
  5. No data-quality gate — no explicit missingness/outlier handling before fitting, which matters a lot for manually-reported district data.
  6. No hierarchical reconciliation — if districts roll up to state totals, forecasts should be made consistent across levels (bottom-up or reconciled), otherwise sums won't match official aggregates.
  7. Naive interval estimation (residual-std normal approx) likely miscalibrates for skewed/count-like data (e.g., case counts, ration distribution numbers).
  8. No drift/retrain-trigger logic — it appears to retrain blindly every run; for an automated agent, periodic blind retraining is wasteful and a drift check is more efficient and more accurate.

On the LLM question specifically

You're right not to let an LLM see raw rows. There are three legitimate non-data-exposing options, in order of risk:

  • Fully local (safest): Replace heuristic thresholds with real statistical diagnostics (STL decomposition strength, ADF/KPSS stationarity, ACF/PACF-based AR order suggestion, coefficient of variation). No LLM needed at all — this alone closes most of the accuracy gap.
  • Metadata-only LLM call (optional, still needs care): Send only an aggregated profile — column types, row count, frequency, seasonal/trend strength scores, missingness %, the deterministic shortlist + their backtest MAEs — and ask the LLM to (a) explain model choice in plain language for non-technical reviewers, or (b) suggest domain-relevant calendar features from column names/context (e.g., "this looks like district rainfall data — consider monsoon-onset indicators"). Never send row-level values.
  • Caution even with metadata: for government data, small aggregate counts (e.g., "3 cases in block X") can themselves be disclosive. Add a k-anonymity-style threshold so any aggregate below a minimum count is suppressed/bucketed before it's ever eligible to leave the device, LLM or not.

I would not put an LLM in the decision-making loop itself (i.e., letting it pick the model) — keep that deterministic and auditable, since this is government-facing. Use the LLM only as an explainer/suggester on top of metadata, gated behind an explicit flag, with a hard-coded suppression rule upstream of it.

Good catch — SELECT * into memory is the actual blocker here, and a flat 5000-row sample has a real bias problem for time series (depends entirely on table physical order, which in Postgres is usually insertion order — so you might be sampling only the oldest or most recent rows, or only certain series if writes were batched by series). Here's what's actually wrong and how to fix it properly, all without bringing the bulk data into Python memory.

Core problem

You're treating "read for inference" and "read for modeling" as the same operation. They aren't. At crore-scale they need completely different strategies:

  • Inference (schema/profile detection) only needs statistics — types, null rates, distinct counts, min/max dates, parse success rate. All of that can be computed inside Postgres with aggregate SQL, touching zero rows in Python.
  • Modeling needs actual time-indexed values, but almost never needs every raw row — it needs the data at the granularity you're going to forecast at (daily/weekly/whatever). So the aggregation should happen in SQL too, via GROUP BY date_trunc(...), before anything reaches pandas.

Fixes, in priority order

  1. Replace row sampling with SQL-side metadata stats. For each column, run aggregate queries (COUNT(*), COUNT(DISTINCT col), COUNT(*) FILTER (WHERE col IS NULL), MIN/MAX, and for date inference, try col::timestamp cast success rate via a query rather than a pandas parse). This gives you exact answers, not sampled approximations, and costs a sequential scan or index scan, not a network transfer of 5000 rows into memory.
  2. If you truly need a row-level preview (e.g. to eyeball values), use TABLESAMPLE SYSTEM (x) or TABLESAMPLE BERNOULLI (x) instead of LIMIT 5000. LIMIT without ORDER BY returns physically-stored rows, which is not a random sample and is especially bad for time series — you could be sampling one contiguous time window. TABLESAMPLE gives you an actual statistical sample across the whole table efficiently.
  3. For the real modeling load, aggregate in SQL, not in pandas. Once time/target/series columns are confirmed, run something like:
sql
   SELECT date_trunc('day', time_col) AS ts, series_col, SUM(target_col)
   FROM table
   GROUP BY 1, 2
   ORDER BY 1

This turns "crores of rows" into "however many time buckets × series exist" — usually a few thousand to a few hundred thousand rows, which is exactly what the forecasting pipeline wants anyway. This single change is probably 90% of your scale problem.

  1. Use a server-side (named) cursor with chunked fetch for any case where you do need row-level data beyond what aggregation covers — never fetchall() an unbounded query.
  2. Check/require an index on the time column before running date-range or date_trunc heavy queries; if missing, the agent should warn and suggest creating one, since this is the difference between seconds and hours at this scale.
  3. The aggregation-feature prompt you want ("aggregate count of column A grouped by column B") should run as a SQL COUNT(*) GROUP BY pushed to Postgres too, and slot in right where you said — after metadata display, before inference — because the resulting engineered column should then be eligible to be picked up as a candidate target/exogenous feature by the inference step.
  4. Inference quality fix: once you're scoring columns from real SQL stats instead of a 5000-row guess, also score date columns by actual parse-success-rate over the full table (via SQL) rather than pandas infer_freq on a sample — sample-based frequency inference is exactly the kind of thing that breaks on irregular real-world reporting data.

Weighed permission constraints, security risks, and operational oversight requirements

It actually can — there's no technical reason the agent can't run a COPY (SELECT ...) TO 'file.parquet'-style export itself. The reason employees do it manually instead isn't a capability gap, it's a few practical and trust reasons that are worth understanding before you decide whether to automate it:

  1. Permissions. Doing an export usually needs different/broader DB privileges than read-only profiling — sometimes a role with pg_read_server_files or filesystem write access on the DB host, or a separate export/ETL service account. Whoever manages the client's DB often deliberately keeps export capability separate from query access, so that no single tool/credential can both read and exfiltrate large volumes of data unsupervised.
  2. It's still the same expensive scan. A full or even date-bounded export still has to read every row it's exporting — so it doesn't sidestep the "this might be heavy on their server" problem at all. It just relocates where the heavy read happens (still on their server) and adds a write step. If anything, exporting the full table is often a bigger one-time hit than running a few aggregate stats queries, since it's reading and serializing every column you select, not just counting/checking.
  3. Oversight and scoping. When a person does the export manually, they typically decide what columns/date range/rows actually need to leave the database — i.e. minimum necessary data — and that decision usually involves someone with authority over what's allowed to leave the client's environment. If your agent silently triggers its own export, that decision gets made by code instead of a person, and on sensitive data that's exactly the kind of thing that should have an explicit human "yes."
  4. Where the file lands matters. Once parquet exists, it's a flat copy of sensitive data sitting somewhere — and now that file's storage, access, and retention become a security question too. A manual export usually happens with someone consciously choosing a secure destination and access control; an automated export needs the same care or it just creates a new sensitive artifact nobody signed off on.

So: it's not "too clever for the agent" — it's that the export step is the moment where permission and scope decisions get made, and right now those decisions belong to a person, not your code. You could add an opt-in --export-parquet flag that runs COPY ... TO PARQUET with a date bound, but it should require the same explicit confirmation and the same "have I actually been told I can do this" check as the SQL profiling does — not become the new quiet default.

Practically, for your situation: ask whoever already does these exports manually whether they would run it for you (so the access/credentials stay with them), or whether you're meant to be granted that same export capability yourself. That answer tells you whether automating it is appropriate at all, separate from whether it's technically easy.
